# 회계법인 취업 허브 사용설명서 — 웹 대시보드 운영

채용공고·업계뉴스·빅펌 인사이트를 자동 수집해 **웹페이지(GitHub Pages)**로 보여주는 정적 대시보드입니다.
**서버도 노트북도 켜 둘 필요 없습니다.** GitHub Actions가 주기적으로 수집→`docs/data/*.json` 커밋→
GitHub Pages가 서빙합니다. (예전 카톡 알림봇과 달리, 내 PC가 꺼져 있어도 자동으로 갱신됩니다.)

---

## ✅ 한눈에

- **보는 곳**: GitHub Pages 주소(저장소 Settings → Pages에 표시되는 URL).
- **갱신 주체**: GitHub Actions(클라우드). 내 PC·카카오톡·화면잠금과 **무관**.
- **내가 할 일**: ① 가끔 사이트가 잘 갱신되는지 확인 ② 카나리아가 Draft PR을 올리면 검토·머지
  ③ 검색 분야/주기 바꾸고 싶을 때 설정 수정.

## ▶ 보기 / 배포

### 사이트 접속
- 저장소 **Settings → Pages** 에 뜨는 URL로 접속(모바일도 반응형).

### GitHub Pages 켜기(최초 1회)
1. GitHub에 push.
2. 저장소 **Settings → Pages → Source: Deploy from a branch, Branch: `main` / `/docs`**.

### 자동 갱신 워크플로 (`.github/workflows/`)
| 워크플로 | 대상 | 주기 |
|---|---|---|
| `run-all.yml` | **채용+기사+인사이트 일괄(주 경로) + 푸시 발송(notifier)** | 외부 핑거(cron-job.org) 30분 |
| `monitor.yml` | **통합 점검**(canary 급감 + sitecheck 신선도 셀프힐링) | **5시간**(cron) |
| `scrape.yml` | 채용공고 | **수동 전용**(cron 제거 — run-all과 중복) |
| `scrape-news.yml` | 기사 | **수동 전용**(cron 제거 — run-all과 중복) |
| `scrape-insights.yml` | 빅펌 인사이트(Chromium 렌더) | **수동 전용**(cron 제거 — run-all과 중복) |
| `canary.yml` | 자기검증 카나리아 | **수동 전용**(cron 없음 — 하루 1회 직접 Run workflow) |
| `freshness.yml` / `sitecheck.yml` | 신선도·라이브 종단 점검 | 매시간 / 3시간 (monitor 안정화 후 cron 폐기 예정) |

- **안정 주기 실행**: GitHub 무료 cron은 자주 드롭돼, 외부 핑거(cron-job.org)가 `repository_dispatch`로 `run-all.yml`을
  30분마다 호출하는 것이 **주 수집 경로**. 개별 `scrape*.yml` cron은 보조.
- **즉시 갱신**: 저장소 **Actions 탭 → 해당 워크플로 → Run workflow**.
- **주기 변경**: 외부 핑거 주기(cron-job.org) 또는 각 파일의 `cron:` 한 줄 수정 후 push.

### 로컬에서 확인(선택)
```
pip install -r requirements.txt
python -m playwright install chromium     # 인사이트(JS 렌더)용 1회
python -m src.export                       # docs/data/{jobs,news,insights}.json 생성
#   부분만:  python -m src.export --part jobs|news|insights
cd docs && python -m http.server 8000      # http://localhost:8000
```

## ⚙ 설정 바꾸기 (`config.yaml`)

- **자격요건(수습CPA/자격무관)·채용구분(인턴/정규/계약/파트)·검색 키워드, 검색 대상, 필터**: `config.py`의 `dashboard` 규칙(키워드 기반, LLM 미사용).

- **수집 주기**: 위 워크플로의 `cron:`.
- **뉴스/인사이트 보존 기간·노이즈 제외어**: `config.yaml`.

> 분류·필터·큐레이션은 전부 키워드 규칙입니다. 하드코딩하지 말고 `config.yaml`에서 조정하세요.

## 🎓 빅4 신입 공채 특집 갱신 (수동 큐레이션)

'방금 올라온 공고' 옆 **빅4 신입 공채** 책갈피에 뜨는 정기채용 일정은 **수동 관리**입니다(자동수집 아님 — 삼일은 ATS 미수집, 정기채용은 시즌성).

- 파일: **`docs/data/big4_recruit.json`** (수집 파이프라인이 건드리지 않음 — `export.py`와 무관).
- 법인별 필드: `firm`(삼일/삼정/안진/한영)·`label`·`title`·`url`·`status`(`open` 접수중 / `upcoming` 업로드 예정 / `closed` 마감)·`tracks`(감사/비감사/파트타임 분리 일정 `[{name,start,end,url}]`; 트랙별 `url`이 있으면 그 줄을 **개별 링크**로 노출, 없으면 박스 전체가 `url`로 이동. 분리 없으면 1개, 일정 미정이면 빈 배열).
- **진행중 아닌 법인**(`status`≠`open`)은 자동으로 **회색(불 꺼진 느낌)** 처리됨.
- 갱신: 시즌마다 각 법인 공식 채용 페이지를 보고 일정·링크를 직접 고쳐 커밋(push)하면 Pages에 반영. **`ensure_ascii=False`**(한글 그대로).

## 📌 ATS 미수집 공고 수동 추가 + 알림 (`docs/data/manual_jobs.json`)

삼일PwC 정기채용처럼 **개별 HTML 페이지라 크롤러가 자동 수집 못 하는 공고**를 **채용목록에 띄우고 푸시 알림까지** 보내려면 이 파일을 씁니다(빅4 특집 박스용 `big4_recruit.json`과 **별개** — 둘 다 채워야 박스+목록 모두 노출).

- 형식: `{"postings": [ {…} ]}`. 항목 필드: `source`(예: `samil`→법인 삼일 자동분류)·`source_label`·**`native_id`(고정 고유키 — 알림 1회 보장)**·`title`·`company`·`deadline`(yyyy-mm-dd)·`posted_date`·`url`·`emp_type`(`정규직`/`파트타임` 등)·`category`(`신입`).
- 동작: `export._load_manual_postings`가 `build_jobs`에 주입 → 크롤 공고와 **동일하게** 분류·NEW 패널·중복제거·`notifier` 푸시. `native_id`가 고정이라 **알림은 처음 1회만**(재발송 없음).
- 시즌 종료/마감 후엔 해당 항목을 **삭제**하면 됨.
- ⚡ **즉시 반영·알림**: 커밋·푸시 후 **GitHub Actions → `run-all` → Run workflow**를 누르면 바로 수집·발송(기다리면 외부 cron 30분 주기).

## 🔔 채용알림 (웹 푸시) 운영

새 공고를 구독자 브라우저로 푸시. 정적 사이트(Pages)라 구독 저장은 **Cloudflare Worker**가, 발송은 **GitHub Actions의 `notifier.py`**가 담당(코어 LLM-free).

**구성**
- 프론트: 검색줄 '🔔 채용알림' → `전체`/`수습CPA 전용` 선택 → `app.js subscribePush()` → `sw.js` 등록 → Worker `/subscribe`(scope 저장).
- 저장소: `worker/`(Cloudflare Worker + KV). 발송: `src/notifier.py`(pywebpush·VAPID 서명), `run-all.yml`이 매 수집 후 호출.
- 설정: `config.py`의 `notifications`(`enabled`·`worker_url`·`vapid_public`). `enabled=true`면 가동.

**시크릿(절대 커밋 금지 — 환경/시크릿에서만, 코드엔 공개키만)**

| 위치 | 키 | 용도 |
|---|---|---|
| GitHub Secrets | `VAPID_PRIVATE_KEY` | 푸시 서명 개인키 |
| GitHub Secrets | `SUBS_READ_TOKEN` | Worker `/list` Bearer (= Worker `READ_TOKEN`과 **동일 값**) |
| Cloudflare(`wrangler secret`) | `READ_TOKEN` | `/list` 인증 |

**셋업 / 재발급 절차**
1. **Worker 배포**: `cd worker && npx wrangler deploy` (최초 1회 `wrangler login` · `wrangler kv namespace create SUBS`로 id 채우기 · `wrangler secret put READ_TOKEN`).
2. **VAPID 키 재발급**(필요 시): `py_vapid`로 키쌍 생성 → **공개키**를 `config.py notifications.vapid_public` + `app.js VAPID_PUBLIC`에 **동일하게**, **개인키**를 GitHub Secret `VAPID_PRIVATE_KEY`로. (공개키 바뀌면 기존 구독 무효 → 재구독 필요.)
3. **GitHub Secrets** 2개 등록(Settings → Secrets and variables → Actions). gh CLI 있으면 `gh secret set`, 없으면 웹 UI.
4. **콜드스타트 억제**: 활성화 직전 1회 `python -m src.notifier --seed`(기존 공고를 baseline 처리 → 가입 직후 폭주 방지) → 커밋된 `state.json`에 반영.
5. `config notifications.enabled=true` + push → 가동.

**점검**: `python -m src.notifier --dry-run`(발송 없이 대상 출력) · Worker `GET /list`(Bearer)로 구독 확인 · `docs/data/notify_status.json`(발송 관측성, freshness가 조용한 발송 실패 감지).

**시험 발송(수동)**: GitHub Actions **`push-test`** 워크플로(workflow_dispatch) → `scripts/push_test.py`가 구독자 전원에게 '시험 알림' 1건 발송(`state` 미변경=멱등, 입력 `body`로 본문 커스텀). VAPID 개인키가 GitHub Secret에만 있어 **로컬 발송은 불가** → 푸시 동작 점검은 이 워크플로로. 결과 로그가 endpoint별 `ok/gone/fail` + 실패 시 HTTP 코드를 출력.

**구독자 수 확인**: ① Worker `GET /list`의 `count`(가장 정확) ② Cloudflare 대시보드 → Storage & Databases → KV → `SUBS`(id `8bcb…`) ③ `wrangler kv key list --namespace-id 8bcb4fecba604ab5bc0c8b203982c8e7 --remote` → ⚠️ **`--remote` 필수**(wrangler 4.x는 기본 LOCAL=빈 미니플레어를 읽어 `[]`로 오인). `sub:` 키 개수 = 구독자 수.

**브라우저별 푸시 서비스(트러블슈팅)**: 같은 PC라도 **크롬=FCM(`fcm.googleapis.com`)·엣지=WNS(`*.notify.windows.com`)·사파리/iOS=Apple**로 엔드포인트가 다름 → 기기마다 별개 구독. **알림이 안 올 때**: ① 구독이 라이브 VAPID 공개키로 됐는지(키 재발급 후 옛 구독은 stale — `app.js`가 키 다르면 자동 재구독하나, 안 되면 사이트에서 알림 껐다 켜기) ② push-test 로그가 해당 endpoint에 `ok`인데 안 뜨면 = OS/브라우저 표시 문제(Windows 집중지원·앱별 알림 허용·엣지 백그라운드 실행) ③ **인앱 브라우저(카톡·네이버앱·인스타 등)는 웹 푸시 미지원** — 프론트가 진입 시 안내 배너 + 구독 시 '다른 브라우저로 열기' 안내(`app.js inAppBrowserName`/`showInAppNotice`). 크롬·사파리 등 외부 브라우저로 열어야 알림 가능.

## 🛡 자기검증 카나리아 (소스 양식 변경/공고 누락 감지)

스크래퍼는 소스 사이트가 HTML을 바꾸면 **조용히 0건/누락**이 납니다. `src/canary.py`가 감시합니다.
**자동 cron 없이 수동 전용** — Actions 탭에서 하루 1회 직접 `Run workflow`로 실행하세요(자동화 미사용은 의도된 설정).
- **구조 체크(무료·LLM 없음)**: 어제 대비 0건/급감/수집실패를 `canary_state.json` 기준으로 감지.
- **시각 체크(LLM, 키 있을 때만)**: 목록 페이지 스냅샷을 Claude vision으로 보고 **신입/수습이 지원 가능한 공고 수**·
  양식 정상여부를 스크래퍼 카운트와 대조. ⭐ 이 사이트가 **의도적으로 경력 공고를 거른다는 큐레이션 의도**를
  프롬프트에 주입(`_project_context`)해 '신입 관점'으로 판정 — 그래서 라이브 페이지에 경력 공고가 더 많아도
  거짓 '누락' 경보가 뜨지 않습니다. 리포트 표의 "화면-신입(LLM)" 열이 이 값입니다.
- **출력물 의도 점검(무료·결정론)**: `filter_leakage`(채용 목록에 경력 전용 공고가 새어들었는지)를 함께 점검해 리포트에 표시.
  (v1.09: 인사이트 '금일' 개념 폐기로 `insights_order` 점검은 제거.)
- **드리프트 시**: `canary_report.md`(진단 + 수정 *제안*)를 담은 **Draft PR 자동 생성**.
  - ⚠️ **자동 머지·자동 프로덕션 커밋은 하지 않습니다.** Claude Code로 검토·보완 후 **사람이 머지**
    (Human-in-the-loop). LLM은 제안만, 확정은 회계사 본인.

### LLM 시각 체크 켜기
- 저장소 **Settings → Secrets and variables → Actions → New repository secret** 에
  `ANTHROPIC_API_KEY` 추가. 없으면 **구조 체크만**(100% 오프라인) 동작.
- 전송 대상은 **공개 채용 페이지 스냅샷뿐**(사적 데이터 없음). API 키는 코드/exe에 절대 내장 금지, secret만 사용.
- **실행 방법**: Actions 탭 → `canary` → **Run workflow**. (cron 자동 실행은 비활성 — 하루 1회 직접 누르는 방식.)
  키 있으면 LLM 시각·신입 카운트 점검까지, 없으면 구조+의도 점검만 동작.

### 뉴스 임베딩(선택) 켜기
- 같은 secret 화면에서 **`VOYAGE_API_KEY`** 추가하면 기사 수집에 임베딩이 켜집니다 — ① 같은 사건 중복 묶기(의미 군집),
  ② 무관 기사 관련성 컷, ③ 카테고리 오분류 보정. **없으면 키워드/어휘 규칙만으로 동작**(100% 오프라인, 안전).
- 비용은 제목만·캐시·게이트라 사실상 무료(대개 free tier). 임계값(`news_embed_*`)은 첫 실행 후 분포 보고 조정.

## 🔧 문제가 생기면

- **사이트가 안 갱신됨**: Actions 탭에서 해당 워크플로의 최근 실행이 **성공(녹색)**인지 확인 →
  실패면 로그 확인. 일시 오류면 **Run workflow**로 재실행.
- **특정 소스만 0건/급감**: 카나리아 Draft PR(`canary_report.md`)을 보면 어디가 어떻게 깨졌는지 진단됨 →
  Claude Code로 해당 어댑터(`src/adapters/*`) 보정 후 머지.
- **페이지는 뜨는데 데이터가 비어 있음**: `docs/data/*.json`이 최근 커밋됐는지 확인(Actions 권한/커밋 실패 점검).
- **검색 분야 조정**: `config.yaml`의 `dashboard` 키워드 수정 후 push.

> 견고성 원칙: 한 소스가 깨져도 나머지는 정상 출력됩니다(`base.safe_fetch`). 전체가 동시에 비는 일은 드뭅니다.

## (보류) 카카오톡 자동 게시 — 레거시

초기 계획은 채용공고를 카톡 오픈채팅방에 자동 게시하는 **알림봇**이었습니다.
카톡 PC GUI 자동화(`src/run.py`·`src/kakao_pc.py`·`src/messenger_bot.js`)가 불안정해(화면잠금·창 포커스·

## 💡 활용 아이디어 (스펙쌓기 — 회계×개발)

지금의 **6개 소스 스크래퍼 + 어댑터 구조**를 그대로 두고 확장할 수 있는 방향:

1. **텔레그램/디스코드 봇** — 같은 스크래퍼 → 공식 봇 API(무료·합법·안정). "RPA 대신 API를 택한 판단"이
   오히려 강점. 클라우드 상주(노트북 불필요).
2. **이메일 일일 다이제스트** — SMTP로 매일 1통. 가장 단순·안정.
3. **채용 데이터 분석** — 공고를 DB에 누적 → 법인별·시즌별·분야별 채용 트렌드 분석(pandas/노트북).
   회계사 도메인 지식 + 데이터분석 = 차별화 포인트.
4. **DART 전자공시 스크래퍼로 확장** — 같은 스크래핑 역량을 감사보고서/공시 수집에 적용 →
   [[감사조서 자동화]] 프로젝트와 시너지.

## 🧰 재사용 가능한 도구/자산

- **소스별 어댑터 패턴** (`src/adapters/`, `src/record.py`) — 사이트마다 다른 HTML/JSON을 공통 레코드로
  수렴. 새 소스는 어댑터 1개만 추가.
- **`src/http_util.py`** — 세션 재사용·재시도·인코딩 보정·커스텀 헤더/JSON 바디.

- **상태/중복제거** (`src/state.py`) — 키 기반 JSON 영속.
- **헤드리스 렌더링** (`src/render.py`) — JS(SPA) 사이트를 Playwright로 렌더 후 링크 추출(인사이트·카나리아 스냅샷).

  - KICPA: `list.face` 정적 + 상세에서 마감일 보강
  - 삼정KPMG: `search.hr` POST → HTML 조각
  - 안진: WiseRecruit2(ASP.NET) 정적 테이블
  - 한영: recruiter.co.kr **JSON API**(`/position/v1/jobflex`, `prefix` 헤더)
  - 삼일PwC: experienced.html Grade 컬럼 + 봇 UA 403 회피
